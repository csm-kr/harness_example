**harness_framework 템플릿을 새 프로젝트로 인스턴스화하는 부트스트래퍼다.**

bootstrap은 단순히 docs 몇 개를 채우는 스킬이 아니다. **빈 슬레이트의 사용자와 대화하며 의도·제약·위험을 끌어내고**, 그것을 다음 모든 곳에 **세팅·검증·패치**한다:

- **헌법** (`CLAUDE.md`) — 플레이스홀더를 사용자 답으로 교체. CRITICAL 규칙 능동 도출.
- **시스템 R&R** (`docs/ROLES.md`, `docs/HOOKS.md`) — 정본 그대로 유지
- **도메인 문서** (`docs/PRD.md`, `ARCHITECTURE.md`, `ADR.md`, 유형별 추가 문서) — 신규 작성
- **스킬** (`.claude/skills/harness.md`, `docker-init.md`, `review.md`) — 정본 그대로 유지(존재 확인)
- **훅 설정** (`.claude/settings.json`) — 정본 그대로 유지(존재 확인)
- **스크립트** (`scripts/execute.py`, `validate.py`, `test_*.py`, `run_stop_hook.sh`, `docker-entrypoint.sh`) — 정본 그대로 유지(존재 확인)
- **Phase 인덱스** (`phases/index.json`) — 빈 배열로 신규 또는 보존
- **환경변수 / 라이선스 / README / GitHub 자산** — 사용자 답에 따라 신규 또는 패치
- **Dockerfile / docker-compose.yml** — 샘플로 두고 다음 단계(`docker-init`)가 교체

bootstrap은 그래서 다음 단계로 작업한다: **점검 → 발견 대화 → 제약 매핑 → 유형 결정 → 정보 수집 → 위험/CRITICAL 도출 → 파일 작성 → 자체 검증 → 인계 요약**.

```
bootstrap   →  docker-init   →  harness   →  execute.py
(인스턴스화)   (컨테이너)        (계획)        (실행)
```

다음 단계와의 책임 분리:
- **Dockerfile/compose 생성** → `docker-init`. bootstrap은 정보 요약만 인계.
- **phase/step 설계** → `harness`. bootstrap은 `phases/index.json` 빈 배열까지만 + **첫 phase 후보 제안**.
- **실제 코드 작성** → `execute.py`가 부르는 Claude 세션. bootstrap은 한 줄도 안 쓴다.

> 시스템 전체의 컴포넌트별 R&R(스킬·훅·docs·phases·스크립트·테스트)은 [`docs/ROLES.md`](../../docs/ROLES.md)에 정리돼 있다. 이 스킬을 실행하기 전에 한 번 읽어 책임 경계를 확인하라.

## 핵심 원칙: 다각적 관점 (Multi-Lens)

bootstrap이 사용자와 대화할 때 **6가지 시각**으로 프로젝트를 본다. 한 가지 시각만으로는 좋은 하네스를 만들 수 없다. 각 단계에서 누락된 시각이 있으면 짚어내라.

| 렌즈 | 묻는 핵심 | 빠지면 생기는 문제 |
|------|----------|-----------------|
| **제품 (Product)** | 무엇을 왜 만드는가? 어떤 가치를 누구에게 제공하는가? | PRD가 비어있어 다음 세션이 잘못된 기능을 만든다 |
| **사용자 (User)** | 누가, 어떤 맥락·제약 아래 쓰는가? | 시나리오 없이 만들면 잘못된 UX/API가 나온다 |
| **운영 (Ops)** | 누가 배포·유지·모니터링하는가? 장애 시 누가 깨어나는가? | 로깅/알림/배포 절차 누락, 운영 중 폭발 |
| **보안 (Security)** | 무엇이 새면 안 되는가? 어떤 입력을 신뢰할 수 없는가? | 시크릿 누설, 권한 경계 무너짐 |
| **미래 (Future)** | 1년 뒤 어떤 상태이길 바라는가? 어떤 변화에 견뎌야 하는가? | 단기 결정이 장기 부채로 누적 |
| **사람 (Team)** | 혼자/팀/오픈소스? 누가 무엇을 결정하고 누가 코드오너인가? | 협업 자산 누락 또는 과잉 |

bootstrap은 4단계(공통 정보 수집)와 7단계(CRITICAL 도출)에서 이 6렌즈를 **명시적으로 한 번씩 통과**시킨다. 사용자가 어느 렌즈를 비웠는지 추적하라.

## 절차

### 0. 환경 점검 — 인프라 무결성 확인 (먼저 한다)

bootstrap은 다음 인프라 파일이 모두 존재해야 동작한다. 이 파일들은 **사용자가 답한다고 만들어지는 게 아니라** 템플릿 정본이다 — 이미 있어야 한다.

| 카테고리 | 파일 |
|---------|------|
| 헌법(템플릿) | `CLAUDE.md` |
| 시스템 R&R | `docs/ROLES.md`, `docs/HOOKS.md` |
| 스킬 | `.claude/skills/harness.md`, `.claude/skills/docker-init.md`, `.claude/skills/review.md` |
| 훅 | `.claude/settings.json` |
| 스크립트 | `scripts/execute.py`, `scripts/validate.py`, `scripts/test_execute.py`, `scripts/test_validate.py`, `scripts/run_stop_hook.sh`, `scripts/docker-entrypoint.sh` |
| 컨테이너 (샘플) | `Dockerfile`, `docker-compose.yml`, `.dockerignore` |
| Git | `.gitignore` |

위 파일들의 존재를 모두 확인한다. 결과에 따라 분기한다.

#### 분기 1 — 모두 존재 (정상)

곧바로 1단계로 진행한다. 사용자에게 한 줄로 보고: "인프라 정본 ✅ 모두 확인됨, 이제 발견 대화를 시작하겠습니다."

#### 분기 2 — 일부만 존재

손상되었거나 사용자가 일부를 지운 상태다. 사용자에게 어떤 파일이 누락됐는지 표로 보여주고, 다음 중 하나를 묻는다:

- **재클론** — `git clone <harness_framework 레포 URL> /tmp/hf && rsync -a /tmp/hf/{누락 경로} ./` 안내
- **무시하고 진행** — 위험. 누락 파일이 가드레일이라면 다음 세션이 가드 없이 동작.

기본은 재클론 권장.

#### 분기 3 — 빈 디렉토리 (CLAUDE.md조차 없음)

사용자가 빈 디렉토리에서 처음 시작한 상태다. bootstrap 스킬 자체는 어디서 호출됐는가? 두 가지 가능성:

1. **글로벌 스킬로 호출됨** — `~/.claude/skills/bootstrap.md`에 설치된 상태에서 빈 디렉토리에서 호출. 이 경우 사용자에게 다음을 안내:
   ```
   현재 디렉토리에 harness_framework 인프라가 없습니다.
   먼저 다음을 실행하세요:
       git clone https://github.com/<owner>/harness_framework.git .
   완료되면 다시 /bootstrap을 호출하세요.
   ```
   그리고 즉시 종료한다. 이유: 인프라 파일을 인라인으로 생성하면 동기화 부담이 너무 크다 — 정본은 항상 git에서 끌어오는 편이 안전.

2. **이 레포 안에서 호출됨** — 그러면 분기 1(정상)이어야 정상이다. 그렇지 않다면 사용자가 의도적으로 일부를 지운 것 → 분기 2로.

> **정책**: bootstrap은 **인프라 파일을 자체적으로 생성하지 않는다**. 정본은 항상 harness_framework 레포에 있고, bootstrap은 그것이 깔린 상태 위에서 사용자 답으로 **패치**할 뿐. 이유: 같은 파일 내용을 두 곳(스킬 안 + 레포 안)에 두면 동기화가 불가능에 가깝다.

#### 재실행 감지 (rebootstrap)

이미 PRD/ARCHITECTURE/ADR이 채워져 있고 `CLAUDE.md`에 플레이스홀더가 없다면, 사용자는 **이미 한 번 bootstrap을 돌린 상태**다. 사용자에게 다음 중 하나를 묻는다:

- **부분 갱신** — 어떤 항목을 수정/추가할지 지정. 그 부분만 다시 묻고 갱신.
- **전체 재작성** — 기존 docs를 `docs/_archive/{YYYYMMDD}/`로 백업 후 처음부터.
- **취소** — 잘못 호출한 경우.

기본은 부분 갱신. 이유: 한 번 채운 의도를 통째로 날리면 손실이 크다.

### 1. 발견 대화 — 빈 슬레이트에서 의도 끌어내기

> 이 단계는 사용자가 "뭘 만들지 명확히 안다"면 빠르게 통과한다. 그러나 "막연한 아이디어만 있다"면 가장 길어지는 단계다. 절대 건너뛰지 마라.

사용자가 **어느 정도 정리되어 있는지**부터 가늠한다. 첫 메시지를 듣고 다음 4구간 중 어디에 해당하는지 판단:

- **A. 명확 (Clarity)** — "Next.js로 블로그 만들고 싶어. SSG로." → 곧바로 3단계(유형 결정)로.
- **B. 윤곽 (Outline)** — "팀 협업 도구를 만들려고 해. 기능은 대략 채팅이랑 작업 트래킹." → 1-1, 1-2만 짧게.
- **C. 동기 (Motivation)** — "회사에서 회의록이 너무 흩어져서 뭔가 만들어야 할 것 같아." → 1-1, 1-2, 1-3 모두 진행.
- **D. 백지 (Blank)** — "뭔가 코딩 프로젝트 시작하고 싶은데..." → 1-1부터 천천히. 결과가 너무 막연하면 사용자에게 "지금은 시작할 단계가 아닐 수도 있어요"라고 솔직히 말한다.

#### 1-1. 동기 (Why) — 5 Whys lite

다음 두 질문을 **순서대로** 한다:

1. "지금 이 프로젝트를 시작하려는 **계기**는 무엇인가요? (해결하려는 문제 / 충족하려는 호기심 / 외부에서 받은 요청 등)"
2. (사용자 답을 듣고) "그게 해결되면 **무엇이 달라지나요**? (사용자 입장에서 / 본인 입장에서)"

답이 추상적이면 ("효율을 높이고 싶다") 한 번 더 파고든다: "구체적으로 지금 어떤 비효율을 겪고 계신가요?"

#### 1-2. 사용자 (Who) + 시나리오

"이 프로젝트의 **첫 사용자**는 누구인가요? 그 사람이 **언제, 어떤 상황에서** 이걸 쓰나요?"

답에서 다음을 추출:

- 1차 사용자가 본인인가, 타인인가
- 동시 사용자 수의 대략적 규모 (1명 / 수십 / 수만)
- 사용 빈도 (한 번 / 매일 / 백그라운드 상시)

이 답으로 이후 **확장성·인증·모니터링** 결정이 강제된다. 사용자가 모르겠다고 하면 **가장 보수적인 가정**을 제안하고 동의를 구한다 (보통 "지금은 본인 + 매일 사용 + 1명").

#### 1-3. 성공의 모양 (What success looks like)

"이 프로젝트가 **3개월 뒤 잘 되었다**고 판단하는 기준은 무엇인가요? (구체적인 숫자/상태/행동)"

이 답이 **PRD의 비기능 요건과 ADR의 결정 기준**이 된다. 답이 "잘 동작하면 좋겠어요" 수준이면 더 파고든다: "어떤 사용자 행동을 보면 잘 됐다고 느끼시겠어요?"

#### 1-4. 미래 시각 (Future lens)

"1년 뒤 이 프로젝트가 **어떤 모습이길** 바라세요? (예: 혼자 쓰는 도구로 그대로 / 팀에 공유 / 오픈소스 공개 / 상용 서비스)"

이 답이 **협업 형태·라이선스·문서 깊이**를 결정한다.

#### 1-5. 발견 결과 요약

위 4가지 답을 **2-3문장으로 요약**해 사용자에게 보여주고 "이 이해가 맞나요?" 확인한다. 어긋나면 어디가 어긋나는지 묻고 보정. 이유: 이후 모든 결정의 기반이라 처음에 정확히 합의하는 비용은 매우 싸다.

---

### 2. 제약 매핑 — 결정을 강제하는 외부 조건

발견 단계가 "원하는 것"을 다뤘다면, 제약 매핑은 "선택할 수 없는 것"을 다룬다. 다음을 **순서대로 하나씩** 묻는다.

1. **마감 / 시간 예산** — 언제까지 어떤 상태여야 하는가? "없음", "느슨한 한 달", "엄격한 3주" 등.
2. **사람 예산** — 혼자인가? 본인 외 누가 코드를 만지는가? (외부 외주 포함)
3. **금전 예산** — 인프라/유료 서비스에 쓸 수 있는 월 비용 한도. 0원이면 0원이라고 답해도 좋다.
4. **컴플라이언스 / 법적 제약** — 다루는 데이터에 PII / 의료 / 금융 / 미성년자 데이터가 포함되는가? 회사 정책상 외부 SaaS 금지가 있는가?
5. **운영 환경 제약** — 배포 대상이 있는가? (자체 서버 / 클라우드 / 모바일 스토어 / 사내망 등)

각 답이 **어떤 결정을 강제하는지** 즉시 사용자에게 보여준다 (한 줄씩). 예시:

- "마감 3주 + 혼자" → "범위(scope)를 좁혀야 해요. PRD의 **Out of scope**를 적극적으로 채울 거예요."
- "PII 포함" → "ADR에 **데이터 저장/암호화 결정**이 들어가고, `docs/SECURITY.md`가 필요해요."
- "월 0원" → "유료 SaaS 의존을 ADR에서 명시적으로 거부할 거예요. (예: 인증 서비스 → 자체 구현 또는 무료 티어)"

답이 명확히 안 나오는 항목은 `TBD`로 두되, **어떤 결정을 후속으로 강제하는지**는 기록해둔다 (CLAUDE.md의 주석으로).

---

### 3. 작업 유형 결정

발견·제약을 종합해 다음 유형 중 하나를 **bootstrap이 먼저 추론해 제안**한다 — 사용자에게 백지로 묻지 마라.

1. **웹/앱 프론트엔드** — Next.js, React, Vue, SvelteKit, Flutter 등
2. **AI/ML 모델 개발** — PyTorch / TensorFlow / JAX 기반 학습·추론
3. **백엔드 API 서버** — FastAPI / Express / Spring / gRPC 등
4. **CLI 도구·라이브러리** — npm / pip / cargo / Homebrew로 배포

추론 휴리스틱:
- "사용자가 브라우저로 접근" → 1 또는 3+1
- "모델 학습 / 데이터셋" → 2
- "다른 시스템이 호출" → 3
- "터미널에서 명령" → 4

다음 형식으로 사용자에게 제시:

> "발견 대화에서 'X'라고 하셨고, 'Y' 시나리오니 **{유형}**이 맞아 보입니다. 다른 의견 있으세요?"

#### 혼합형 처리

"백엔드 + 프론트엔드", "CLI + 라이브러리" 같은 혼합은 흔하다. 이 경우:

- **하나를 주(主)로** 정한다 (먼저 만들 것 / 더 핵심인 것).
- 나머지는 **부속**으로 처리하고, 그 유형의 도큐먼트 일부만 추가한다.
- 예: "주: 백엔드 API + 부: 관리용 CLI" → 4-1 공통 + 3-C 백엔드 풀세트 + 3-D 중 `API_REFERENCE.md`만.

#### 어디에도 안 맞는 경우

데이터 엔지니어링, 데스크톱 앱, 게임, 펌웨어 등은 4가지 외 변형이다. 이 경우:

1. 가장 가까운 유형을 고른다 (대부분 백엔드 또는 CLI에 가깝다).
2. **빼야 할 산출물**을 사용자와 같이 결정한다. 예: 데이터 엔지니어링 → API_SPEC.md 대신 `docs/PIPELINE.md` 신규.
3. 추가 도큐먼트 1-2개를 신규 정의 (이름·섹션 윤곽). 만들지는 말고 사용자 합의만.

### 4. 공통 정보 수집

**한 항목씩 순서대로** 묻는다. 한 번에 모아 묻지 마라. 이유: 답이 산만해지고, 앞 답에 따라 뒤 질문이 달라지는 경우가 많다.

각 항목마다 **나쁜 답 vs 좋은 답 예시**를 머릿속에 두고, 사용자 답이 나쁜 답에 가까우면 한 번 더 파고들어라.

#### 4-1. 프로젝트명

- 디렉토리·이미지·컨테이너명에 쓰인다. **kebab-case 권장**.
- 좋은 답: `team-notes-v2`, `wiki-rag-bot`
- 나쁜 답: `MyProject`, `temp123`, `테스트` (한글은 컨테이너명에 못 씀)

#### 4-2. 한 줄 목적

- PRD의 첫 문장으로 그대로 들어간다.
- 좋은 답: "사내 회의록을 자동 요약해 검색 가능한 위키로 만든다."
- 나쁜 답: "효율을 높인다", "AI 도구"
- 안티패턴: 동사가 없는 답, 대상이 없는 답, 형용사("좋은", "효율적인")만으로 채운 답.

#### 4-3. 핵심 기능 3가지

- 우선순위 순서.
- "3가지 이상이 필수"라고 사용자가 말하면, **MVP에서 뺄 1-2개**를 골라달라고 다시 묻는다. 이유: 3개 넘는 핵심은 핵심이 아니다 — 우선순위가 안 잡혀있다는 뜻.
- 답을 들으면 곧바로 **각 기능의 1차 사용자 시나리오**를 한 줄씩 같이 받는다 (PRD에 들어감).

#### 4-4. 기술 스택

- 언어/프레임워크/주요 라이브러리. **버전까지** 받는다.
- 답이 막연하면 ("Python으로") → 발견 단계의 답으로부터 합리적 추천을 1-2개 제시. 예: "AI/ML + GPU 사용 → Python 3.12 + PyTorch 2.x 추천. 다른 의견 있으세요?"
- **각 굵은 결정마다 ADR 1항목**이 생긴다는 것을 사용자에게 미리 알린다.

#### 4-5. 라이선스

- MIT / Apache-2.0 / 비공개(proprietary) / 기타.
- 잘 모르면 1-4의 답을 본다: "혼자 쓰는 도구"면 비공개 추천, "오픈소스 공개"면 MIT 추천.
- 잘못된 OSS 라이선스보다 비공개가 안전. 결정 못 하면 비공개로 두고 README에 한 줄.

#### 4-6. 협업 형태

- 단독 / 소규모 팀 / 오픈소스. (1-4 미래 시각 답과 일치하는지 확인)
- 이 답으로 GitHub 자산(`.github/`, `CODEOWNERS`) 생성 여부가 갈린다.

#### 4-7. 6렌즈 누락 점검

여기까지 받은 답으로 **6렌즈 중 어느 것이 비어있는지** 사용자에게 보여준다:

```
[6렌즈 충족도]
- 제품 (Product):  ✅ 한 줄 목적 + 핵심 기능
- 사용자 (User):    ⚠️ 시나리오는 있으나 동시 사용자 규모 TBD
- 운영 (Ops):       ❌ 누가 배포·모니터링하는지 미확인
- 보안 (Security):  ⚠️ PII 포함 여부만 확인. 권한 경계는 미정.
- 미래 (Future):    ✅ 1년 뒤 그림 있음
- 사람 (Team):      ✅ 단독
```

⚠️/❌가 있는 항목은 다음 5/6단계에서 채울 기회를 준다. 사용자가 "지금은 거기까지 정하고 싶지 않다"고 하면 TBD로 둔다 — 강요하지 마라.

### 5. 유형별 추가 정보 수집

선택한 유형에 따라 다음을 추가로 묻는다.

#### 5-A. 웹/앱 프론트엔드

- UI 라이브러리 (Tailwind / MUI / ShadCN / 자체 디자인 시스템)
- 라우팅 방식 (파일 기반 vs 코드 기반)
- 인증 필요 여부 + 방식 (OAuth / 세션 / JWT / 없음)
- **렌더링 모드** (SSR / SSG / CSR / 혼합) — Next.js 류는 결정 강제
- **타깃 디바이스** (데스크톱 / 모바일 우선 / 둘 다) — 디자인 토큰 단위 결정

#### 5-B. AI/ML 모델 개발

- GPU 사용 여부 + CUDA 버전 (없으면 CPU)
- 학습 vs 추론 비중 (학습 위주 / 추론 위주 / 둘 다)
- 실험 추적 도구 (TensorBoard / W&B / MLflow / 없음)
- 데이터셋 소스 (로컬 / S3 / HuggingFace / 자체 수집)
- **재현성 요건** — 시드 고정 의무? 데이터 버전 관리 도구(DVC 등) 사용?
- **모델 배포 형태** — 학습만 / 추론 API / 임베디드 / 결정 안 함

#### 5-C. 백엔드 API 서버

- 프로토콜 (REST / gRPC / GraphQL / 혼합)
- DB (PostgreSQL / MySQL / MongoDB / Redis 등 + 버전)
- 인증/인가 (JWT / 세션 / OAuth / API Key)
- **상태성** (stateless / 세션 보관 / 워크플로우 보관)
- **비동기 작업 큐 필요 여부** (Celery / RQ / Sidekiq 등)

#### 5-D. CLI 도구·라이브러리

- 배포 채널 (npm / pip / cargo / Homebrew / 직접 바이너리)
- 공개 API 표면 (함수 단위 / 클래스 단위 / 서브커맨드 단위)
- 호환 대상 (예: Python 3.10+, Node 20+)
- **상태 저장 위치** (stdin/stdout만 / 사용자 홈 / 시스템 디렉토리)
- **외부 네트워크 접근 여부** — 오프라인 동작 보장이 필요한가?

답하기 어려운 항목은 `TBD`로 두되, 이후 ADR/유형별 문서에 "TBD — 결정 필요" 한 줄로 남긴다.

### 6. 위험·실패 모드 매핑 (가벼운 위협 모델링)

**작은 프로젝트라도 위험 매핑은 한 번은 해야 한다.** 하지 않으면 다음 세션이 뭘 보호해야 하는지 모른다.

다음 4가지 위험 카테고리를 사용자에게 한 줄씩 묻는다. 답이 "해당 없음"이면 그것도 명시적으로 기록한다.

| 위험 카테고리 | 질문 | "있음" 답이 강제하는 것 |
|------------|-----|---------------------|
| **데이터 손실** | 잃으면 복구 불가능한 데이터가 있는가? | 백업 정책, 마이그레이션 정책 (`MIGRATIONS.md`) |
| **시크릿 누설** | API 키 / 토큰 / DB 자격증명을 다루는가? | `.env.example` 강화, `.gitignore` 점검, SECURITY.md |
| **잘못된 출력** | 사용자에게 잘못된 결과를 주면 어떤 피해가 있는가? | 평가 프로토콜 (AI/ML), 입력 검증 정책 (백엔드) |
| **다운타임** | 몇 시간 죽으면 누구한테 어떤 영향이 가는가? | 모니터링/알림 결정, SLO 명시 |

각 카테고리마다:
- "있음" → 어떤 문서·결정이 강제되는지 사용자에게 보여주고, 7단계의 CRITICAL 후보로 접수.
- "없음/모르겠음" → 그 사실을 PRD의 "비기능 요건" 섹션에 "현재 보호 대상 아님"으로 명시.

> **핵심 통찰**: 위험을 인지했는데 결정 안 함 ≠ 위험을 모름. 인지된 위험은 문서에 명시된다 — 다음에 이 프로젝트를 이어받는 사람이 알아야 한다.

### 7. CRITICAL 규칙 도출 — 능동적 elicitation

`CLAUDE.md`의 CRITICAL 규칙은 **bootstrap의 가장 중요한 결과물**이다. 단순히 플레이스홀더를 교체하는 게 아니라, **사용자에게 직접 질문해 도출**한다.

#### 7-1. 안티패턴 차단

다음 두 함정을 미리 사용자에게 알린다:

- **모든 것이 CRITICAL이면 아무것도 CRITICAL이 아니다.** 3-5개로 제한하라.
- **CRITICAL은 "X를 하라"가 아니라 "Y를 하지 마라"에 더 가깝다.** 위반하면 즉시 알려야 하는 강제 규칙.

#### 7-2. 6렌즈 통과 — CRITICAL 후보 끌어내기

다음 질문을 한 항목씩 던진다. 각 렌즈마다 **0-2개**의 후보가 나오면 충분하다.

- **제품**: "사용자에게 절대 보여주면 안 되는 상태/메시지가 있나요?" (예: 생 SQL 에러, 내부 디버그 로그)
- **사용자**: "사용자가 절대 잃어선 안 되는 입력/상태가 있나요?" (예: 작성 중인 글, 결제 직전 카트)
- **운영**: "프로덕션 배포 전 반드시 통과해야 하는 절차가 있나요?" (예: 마이그레이션 dry-run, 테스트 통과)
- **보안**: "어떤 데이터/액션이 권한 검사 없이 일어나면 안 되나요?" (예: 사용자 데이터 조회, 외부 API 호출)
- **미래**: "한 번 정해지면 바꾸기 어려운 결정이 있나요? 어떤 변화에 견뎌야 하나요?" (예: DB 스키마 변경 정책, API 버전 호환성)
- **사람**: "팀원이 새로 합류하면 첫날 반드시 알아야 하는 규칙이 있나요?" (예: 어떤 PR도 리뷰 없이 merge 금지)

#### 7-3. 후보 정제

후보가 5개 넘으면 사용자와 함께 **CRITICAL vs 일반 규칙**으로 분류한다.

- CRITICAL = 위반 시 즉시 차단/롤백 대상
- 일반 = 권장사항, 코드 리뷰에서 지적 대상

`CLAUDE.md`의 `## 아키텍처 규칙` 섹션에 다음 형식으로 적는다:

```markdown
- CRITICAL: <규칙 한 줄>. 이유: <왜>. 위반 시: <어떻게 알 수 있는가 / 무엇을 해야 하는가>.
- <일반 규칙 한 줄>
```

**"이유"와 "위반 시"를 빼지 마라.** 다음 세션이 규칙의 적용 범위와 강제력을 판단하려면 이 두 줄이 꼭 필요하다.

#### 7-4. 개발 프로세스 규칙

`## 개발 프로세스` 섹션은 다음 후보 중 사용자와 함께 고른다:

- TDD 의무 여부 (`기본 ON` — 끄려면 명시적 결정)
- 커밋 메시지 컨벤션 (Conventional Commits 권장)
- 브랜치 네이밍 규칙
- PR 리뷰 의무 (협업 형태가 팀/오픈소스인 경우)

각 항목은 CRITICAL 또는 일반으로 분류해 박는다.

#### 7-5. 플레이스홀더 0개 검증

마지막으로 `CLAUDE.md`를 다시 열어 `{...}` 패턴이 **단 한 개도 남아있지 않은지** 정규식으로 확인한다. 남아있으면 사용자와 함께 채운다 — TBD조차 허용하지 않는다 (다른 문서에서는 TBD 허용, CLAUDE.md만 예외). 이유: 이 파일은 모든 세션에 자동 주입되어 다른 모든 결정의 우선순위 1번이다.

### 8. 파일 생성

수집한 정보로 아래 파일을 만들거나 채운다.

**기존 파일이 있으면 덮어쓰기 전에 사용자에게 diff를 보여주고 확인을 받아라.** 이유: 이미 작성된 내용이 있을 수 있다.

각 파일에는 **품질 게이트**(좋은 결과물의 smell test)가 있다. 작성 직후 자체 점검하라.

#### 8-1. 공통 파일 (모든 유형)

##### `docs/PRD.md`

채울 섹션:
- **목적** — 한 줄 목적 + 1~2문단 배경 (1-1 발견 답 활용)
- **사용자 / 사용 시나리오** — 누가, 언제, 어떤 문제로 쓰는가 (1-2 답 활용)
- **핵심 기능** — 4-3에서 받은 3가지를 우선순위 순으로
- **비기능 요건** — 6단계 위험 매핑에서 도출. 성능 / 보안 / 확장성 / 가용성 중 해당하는 것
- **성공 기준** — 1-3 답을 측정 가능한 형태로
- **범위 외(Out of scope)** — 이번 단계에서 하지 않을 것. **반드시 1개 이상 적어라.** 비어있으면 "이 프로젝트는 모든 것을 다 한다"는 거짓 신호가 된다.

**품질 게이트**: 핵심 기능과 범위 외 사이에 명확한 경계선이 있는가? 비기능 요건 중 위험 매핑에서 "있음"이라고 답한 것이 모두 반영되었는가?

##### `docs/ARCHITECTURE.md`

- **디렉토리 구조** — 트리 형태. 유형별 표준 레이아웃을 따른다 (예: 웹은 `app/` `components/`, AI/ML은 `src/` `runs/` `data/`).
- **데이터 / 요청 흐름** — 한 다이어그램 + 짧은 설명. 텍스트 다이어그램으로 충분하다.
- **레이어 책임** — 각 디렉토리/모듈이 무엇을 담당하고 무엇을 하지 않는가.
- **외부 의존** — 외부 SaaS / DB / 큐 / 모델 등 의존하는 것 모두 표로. (운영 렌즈 산물)

**품질 게이트**: 모든 핵심 기능이 디렉토리 구조 안에서 어디에 살지 추적 가능한가? "여기는 무엇을 안 한다"가 한 번이라도 적혀 있는가?

##### `docs/ADR.md`

기술 스택 결정 기록. 각 결정은 다음 형식으로:

```
## ADR-{번호}: {결정 제목}

- **상태**: accepted
- **날짜**: YYYY-MM-DD
- **맥락**: 왜 이 결정이 필요했는가 (제약 매핑/위험 매핑에서 도출된 강제력 인용)
- **결정**: 무엇을 선택했는가
- **대안**: 검토했지만 채택하지 않은 것 + 이유
- **결과**: 이 결정으로 인한 제약/이점
```

**최소 항목**: 언어 1, 프레임워크 1, DB(있으면) 1, 인증(있으면) 1. 위험 매핑에서 강제된 결정은 반드시 ADR 1항목으로.

**품질 게이트**: 각 ADR의 "맥락"이 PRD나 위험 매핑의 어떤 항목을 인용하는가? 인용이 없으면 "왜 그 결정을 했는가"가 미래에 사라진다.

##### `CLAUDE.md`

7단계에서 이미 채웠다. 8단계에서는 단지 **플레이스홀더 0개 검증을 한 번 더** 한다.

`기술 스택` 섹션은 ADR과 일관되게. `명령어` 섹션은 일단 비워두거나 `TBD`로 둔다. 실제 값은 `docker-init`이 결정한다.

##### `phases/index.json`

이미 존재하지 않으면 다음으로 생성:

```json
{
  "phases": []
}
```

이미 있으면 건드리지 마라.

##### `README.md` (프로젝트 README)

현재 `README.md`는 **하네스 뼈대용**이다. 이걸 그냥 덮어쓰면 안 된다. 이유: 뼈대 사용법(Quick Start, FAQ 등)을 잃는다.

절차:

1. 기존 `README.md`가 있고 첫 줄이 `# harness_framework`로 시작하면 → `docs/README.harness.md`로 옮긴다.
2. 그 외에 사용자가 작성한 README가 있으면 → 사용자에게 diff를 보여주고 어떻게 할지 묻는다.
3. 새 `README.md`를 작성한다. 최소 섹션:
   - 한 줄 목적 (PRD에서 가져옴)
   - Quick Start (cp .env, docker compose up, 첫 실행 한 줄)
   - 기술 스택
   - 디렉토리 구조 (ARCHITECTURE.md 링크)
   - 라이선스

##### `LICENSE`

선택한 라이선스의 표준 텍스트를 넣는다. 비공개(proprietary) 선택 시 `LICENSE` 파일 자체를 만들지 말고 `README.md`에 "All rights reserved" 한 줄만 넣는다. 이유: 잘못된 OSS 라이선스 텍스트가 들어가는 것보다 비워두는 편이 안전하다.

저작권자/연도는 사용자에게 물어 채운다. 모르면 `git config user.name` 결과를 기본값으로 제안한다.

##### `docs/SECURITY.md` (위험 매핑 결과에 따라 조건부)

6단계의 "시크릿 누설" 또는 "데이터 손실"이 "있음"이면 신규 작성:

```markdown
# Security

## 보호 대상
- {예: 사용자 PII, API 키, 내부 토큰 등 — 위험 매핑 답에서 가져옴}

## 시크릿 관리
- 저장 위치: 환경변수 (.env). 운영은 {예: AWS Secrets Manager}로 이전 예정.
- 커밋 금지 패턴: .env, *.pem, credentials.json (.gitignore 참조)

## 사고 대응
- 시크릿 누설 의심 시: {연락처 또는 절차}
- 취약점 보고: {보고 채널}
```

위험이 "없음/모르겠음"이면 만들지 말고 `TROUBLESHOOTING.md`에 "현재 보호 대상 없음으로 결정 (YYYY-MM-DD)" 한 줄만 남긴다.

##### 단독 (혼자 쓰는 프로젝트)

GitHub 자산은 만들지 않는다. 이유: 사용하지 않을 템플릿이 레포만 어지럽힌다.

##### 소규모 팀 / 오픈소스

다음을 생성한다:

- `.github/workflows/ci.yml` — push/PR 시 `pytest scripts/` + `python3 scripts/validate.py` 자동 실행. 컨테이너에서 돌리는 게 정석이지만 CI에서는 `actions/setup-python@v5` + 필요한 의존성 설치로 단순화.
- `.github/PULL_REQUEST_TEMPLATE.md` — 변경 요약 / 관련 phase / 검증 방법 / 체크리스트 섹션을 한국어로.
- `.github/ISSUE_TEMPLATE/bug_report.md`, `feature_request.md` — 오픈소스만. 팀이면 생략.

오픈소스 추가:

- `CODEOWNERS` (루트 또는 `.github/`) — 사용자에게 코드 오너 GitHub ID를 묻고 채운다. 모르면 빈 파일 + 주석으로 두고 첫 PR 전에 채우라고 안내.
- `CONTRIBUTING.md` — 기여 절차(브랜치 네이밍, 커밋 컨벤션, PR 절차) 한국어로.
- `SECURITY.md` — 위험 매핑 결과를 반영해 취약점 비공개 보고 채널 한 줄.

##### AI/ML — 다음 4개 파일 신규

`docs/EXPERIMENTS.md`:
- **디렉토리 컨벤션**: `runs/{YYYYMMDD-HHmm}-{tag}/` 아래 `config.yaml`, `metrics.csv`, `checkpoints/`, `logs/`

- **하이퍼파라미터 스냅샷**: 실행 시작 시 `config.yaml`을 `runs/.../config.yaml`로 자동 복사. 코드 git rev도 함께 기록.
- **실험 추적 도구 연동** (선택한 도구가 있으면 그 도구의 init 코드 위치만 명시)

`docs/DATA_CARD.md`:
- 데이터셋 출처·라이선스·취득일

- 알려진 편향과 한계

`docs/MODEL_CARD.md`:
- 모델 의도된 용도 + 권장하지 않는 용도
- 학습 데이터 요약 (DATA_CARD 링크)
- 평가 결과 요약 (EVAL_PROTOCOL 링크)
- 알려진 한계·편향·실패 모드

`docs/EVAL_PROTOCOL.md`:
- 평가 데이터셋 구분 (train/val/test 또는 별도 hold-out)
- 사용할 메트릭과 측정 방식
- 비교 baseline (있다면)
- 통계적 유의성 검정 방식 (시드 여러 개 평균 등)

GPU를 쓴다면 Dockerfile 베이스 이미지를 `nvidia/cuda:{버전}-runtime-ubuntu22.04` 류로 바꿔야 한다는 것을 사용자에게 명시적으로 알리고, `EXPERIMENTS.md`에도 한 줄 적어둔다. 실제 변경은 `docker-init`에서.

##### 백엔드 — `docs/API_SPEC.md` + `docs/MIGRATIONS.md` 신규

`docs/API_SPEC.md`:
- 엔드포인트 표 (메서드 / 경로 / 요청 스키마 / 응답 스키마 / 에러 코드)
- 인증/인가 정책 (어떤 엔드포인트가 어떤 권한을 요구하는가)
- 버전 관리 정책 (URL prefix `/v1/` vs 헤더 `Accept-Version`)
- 페이지네이션·정렬·필터링 컨벤션
- Rate limit 정책 (있다면)

`docs/MIGRATIONS.md`:
- 마이그레이션 도구 (Alembic / Prisma / Flyway / 직접 SQL 등)
- 마이그레이션 파일 위치 컨벤션 (예: `db/migrations/{ts}_{slug}.sql`)
- forward / rollback 정책 (rollback 작성 의무 여부)
- 시드 데이터 위치 + 적용 시점
- 운영 환경 배포 절차 (수동 승인 게이트 등)

##### CLI/라이브러리 — `docs/API_REFERENCE.md` + `docs/MIGRATION.md` + `examples/README.md` 신규

`docs/API_REFERENCE.md`:
- 공개 함수 / 클래스 / 서브커맨드 시그니처
- 각 항목의 입력·출력 예시
- 호환성 정책 (semver 적용 여부, breaking change 정의 — 무엇이 깨지면 major bump인가)
- deprecation 절차 (몇 버전 동안 경고 후 제거)

`docs/MIGRATION.md` (사용자가 버전 간 이동할 때 보는 문서, 위와 다름):
- 비어있는 템플릿. 첫 breaking change가 생기면 채운다.
- 형식: 버전별 섹션 + Before/After 코드 예시 + 자동 마이그레이션 도구가 있다면 명시.

`examples/README.md`:
- 디렉토리 컨벤션: `examples/{번호}-{slug}/` 단위로 자기완결 예제.
- 각 예제는 README + 실행 가능한 스크립트 + 기대 출력을 포함한다.
- 새 공개 API 추가 시 예제도 함께 추가한다는 정책 명시.

#### 8-4. 직접 만들지 말 것 — Docker 환경

**`Dockerfile`, `docker-compose.yml`은 만들지 마라.** 이유: `docker-init` 스킬이 같은 정보를 다시 묻고 한 번 더 만들면 충돌한다.

대신 마무리 단계에서 사용자에게 다음을 출력한다:

- 수집한 기술 스택, GPU 여부, DB, 추가 서비스 등을 **요약 블록**으로 정리
- "다음 메시지에서 `docker-init`을 호출하면 위 요약을 그대로 붙여넣을 수 있도록 준비됐다"고 안내

### 9. 자체 검증 — 끝나기 전 품질 게이트

bootstrap을 끝내기 전 다음 체크리스트를 자체 실행하고 결과를 사용자에게 보여준다. ❌가 있으면 끝내지 말고 보정한다.

```
[자체 검증]
| 항목                                                    | 결과 | 비고 |
|--------------------------------------------------------|------|------|
| CLAUDE.md에 {중괄호 플레이스홀더}가 0개인가              | ✅/❌ | grep '{[^}]*}' CLAUDE.md |
| CLAUDE.md의 CRITICAL 항목이 3-5개 사이인가              | ✅/❌ | 3개 미만 → 도출 부족, 5개 초과 → 분류 필요 |
| 각 CRITICAL이 "이유"와 "위반 시"를 모두 포함하는가      | ✅/❌ | |
| PRD.md의 "범위 외(Out of scope)"가 비어있지 않은가      | ✅/❌ | 비어있으면 모든 것을 한다는 거짓 신호 |
| ADR이 1개 이상이고 각 항목에 "맥락" 인용이 있는가         | ✅/❌ | 인용은 PRD/위험 매핑/제약을 가리켜야 함 |
| ARCHITECTURE.md에 디렉토리 구조 + 외부 의존 표가 있는가  | ✅/❌ | |
| 위험 매핑의 "있음" 항목이 모두 어떤 문서/규칙에 반영됐는가 | ✅/❌ | 6단계의 표를 ADR/SECURITY/PRD와 cross-check |
| 6렌즈 중 ❌ 비어있는 렌즈에 사용자가 명시적으로 동의했는가 | ✅/❌ | TBD라도 의식적으로 결정된 상태여야 함 |
| .env.example에 GIT_USER_NAME/EMAIL이 있는가              | ✅/❌ | |
| phases/index.json이 유효한 JSON이고 phases가 배열인가    | ✅/❌ | |
```

이 결과를 **5단계의 산출물 일람표 직전**에 출력한다. 사용자가 ❌를 무시하고 진행하길 원하면 그 결정을 `docs/TROUBLESHOOTING.md`에 한 줄 남긴다.

#### 10-1. 산출물 일람표 출력 (필수)

각 행에는 **bootstrap이 한 액션**을 표시한다:

- 🆕 **신규 작성** — bootstrap이 사용자 답으로 새로 만든 파일

#### 10-2. Docker 인계 요약 블록

다음 형식으로 출력한다. 사용자가 그대로 복사해 `docker-init` 호출에 붙여넣을 수 있어야 한다.

```
- 운영 제약: {예: 외부 SaaS 금지 / 사내망 배포 / 자유}
```

#### 10-3. 첫 phase 후보 제안 (harness 인계)

bootstrap은 phase를 만들지 않지만, **사용자가 다음 `/harness` 호출에서 무엇부터 만들지 막막하지 않도록 후보 1-2개를 제안**한다. 유형별 휴리스틱:

- **웹/앱**: `0-skeleton` (라우팅 + 레이아웃 + 더미 페이지) → `1-{핵심 기능 1}`
- **AI/ML**: `0-data-pipeline` (데이터 로더 + 전처리) → `1-baseline-model`
- **백엔드**: `0-skeleton` (서버 부팅 + healthcheck) → `1-{핵심 도메인}-crud`
- **CLI**: `0-cli-skeleton` (서브커맨드 골격 + --help) → `1-{핵심 명령} 구현`

다음 형식으로 출력:

```
[harness 첫 phase 제안]
1순위: 0-{slug} — {한 줄 설명}. 이유: 다른 모든 phase의 전제.
2순위: 1-{slug} — {한 줄 설명}. 이유: 핵심 기능 #1의 가장 작은 검증.

선택은 다음 메시지에서 /harness를 호출하면서 알려주세요.
```

## 대화 휴리스틱 — 막힐 때

bootstrap은 **사용자가 답을 못 만들 때**가 가장 어렵다. 다음 패턴을 쓴다.

### 사용자가 "잘 모르겠어요"라고 답하면

1. **선택지 제공** — 두세 개의 합리적 옵션을 짧게 보여준다. 빈 공간을 채워달라고 하지 마라.
2. **가장 보수적인 가정으로 채우기** — 옵션 중 가장 안전한 것을 기본값으로 제안하고 동의를 구한다.
3. **TBD로 두기 (단, 의식적으로)** — 위 둘이 안 되면 TBD. 단 어느 시점에 누가 이 결정을 해야 하는지 함께 적어둔다 ("MVP 첫 사용자가 생기기 전에 ADR-N에서 결정").

CLAUDE.md의 CRITICAL 규칙은 TBD를 허용하지 않는다 (7-5). 다른 곳은 TBD가 OK.

### 답이 너무 추상적이면

다음 한 가지 기법으로 한 번 더 파고든다:
- **5 Whys** — "그게 왜 중요한가요?"를 1-2번 더.
- **반대 사례** — "그게 안 되면 무엇이 나쁜가요?"
- **구체 시나리오** — "오늘 이 도구가 있다고 가정하고, 어떤 순간에 어떻게 쓰시겠어요?"

세 번 시도해도 추상적이면 더 파고들지 말고 TBD로 두고 다음 항목으로. 첫 세션에서 모든 결정을 강제하지 마라.

### 답이 너무 야심차면 (모든 게 핵심, 모든 게 CRITICAL)

다음 두 질문으로 정리한다:
- "셋 중 둘만 만들 수 있다면 무엇을 빼시겠어요?"
- "사용자가 이 중 하나만 본다면 어느 것이 가장 큰 인상을 남겼으면 하시나요?"

### 사용자가 답을 바꾸면

이전 답을 갱신해야 할 곳을 모두 찾아 함께 갱신한다. PRD/ARCHITECTURE/ADR/CLAUDE.md는 서로 참조 관계라 한 곳만 바꾸면 모순이 생긴다. 변경 직전 사용자에게 영향 범위를 한 줄로 보여줘라.

## 주의사항

- **한국어로 질문하라.** 사용자의 글로벌 규칙이다.
- **한꺼번에 묻지 마라.** 한 번에 한 항목씩 받는다. 이유: 답이 산만해지고 앞 답에 따라 뒤 질문이 달라진다.
- **6렌즈를 의식하라.** 4단계와 7단계에서 명시적으로 한 번씩 통과시킨다. 비어있는 렌즈는 사용자에게 보여주고 결정을 받는다.
- **확신 없는 값으로 채우지 마라.** 사용자가 답하지 않은 부분은 `TBD`로 두되, 어느 시점·어느 결정이 강제 트리거가 되는지 함께 기록한다.
- **기존 파일을 묻지 않고 덮어쓰지 마라.** 존재하면 diff를 보여주고 확인받는다. 이유: 사용자가 이미 작성한 내용을 잃을 수 있다.
- **`CLAUDE.md`의 `{중괄호}` 플레이스홀더는 단 하나도 남기지 마라.** 다음 세션이 잘못된 컨텍스트를 읽는다. CLAUDE.md만 TBD 금지 — 다른 문서는 TBD 허용.
- **CRITICAL 규칙은 능동적으로 도출하라.** 7단계의 6렌즈 질문을 빠뜨리지 마라. 단순 플레이스홀더 교체는 가장 큰 안티패턴이다.
- **위험 매핑을 건너뛰지 마라.** 작은 프로젝트라도 6단계는 한 번은 거친다. "해당 없음"도 명시적 답이다.
- **자체 검증을 출력 없이 끝내지 마라.** 9단계의 체크리스트는 모든 세션이 보여줘야 한다.
- **Dockerfile / docker-compose.yml은 만들지 마라.** `docker-init`에 위임. 정보 요약만 사용자에게 전달한다.
- **`/harness`를 직접 호출하지 마라.** bootstrap의 책임이 아니다. 첫 phase 후보 제안과 다음 단계 안내까지만.
- **유형 선택을 강제하지 마라.** 발견·제약 답으로 추론해 제안하고, 사용자가 동의/수정하게 한다.
- **재실행을 지원하라.** 이미 채워진 상태에서 호출되면 부분 갱신 / 전체 재작성 / 취소를 묻고, 절대 묻지 않고 통째로 덮지 마라.
