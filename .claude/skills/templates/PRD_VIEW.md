# PRD Review Views

PRD 검수에서 각 View는 “이 제품 문서를 어떤 렌즈로 의심하고 검토할 것인가”를 의미한다.
아래 View들은 기능 구현 전 PRD 단계에서 제품 방향, 사용자 경험, 기술 구조, 보안, 비용, 성장 가능성을 사전에 점검하기 위한 검수 관점이며, **특정 제품/도메인에 의존하지 않는 일반 템플릿**으로 작성되어 있다.

> 사용법: 각 View의 "주요 검수 질문"을 그대로 사용하고, "도메인 적용 가이드"를 참고하여 검수 대상 제품의 핵심 엔티티/플로우/리소스를 대입해 검수 질문을 구체화한다.

---

## 1. Product Manager / Product Strategist View

### 역할
제품이 왜 존재해야 하는지, 어떤 문제를 해결하는지, MVP 범위가 적절한지를 검수한다.

### 주요 검수 질문
- 이 제품이 해결하는 핵심 문제가 명확한가?
- 타깃 유저가 분명한가?
- MVP 범위가 너무 크지 않은가?
- 지금 꼭 필요한 기능과 나중에 할 기능이 구분되어 있는가?
- 기능들이 하나의 제품 방향으로 수렴하는가?
- 제품의 North Star가 명확한가?

### 도메인 적용 가이드
- 이 제품이 속한 카테고리(저장형/협업형/소비형/거래형 등) 중 어느 쪽인지 정의되어 있는가?
- 첫 버전에서 반드시 검증해야 하는 핵심 가치 가설은 무엇인가?

---

## 2. User Advocate View

### 역할
실제 사용자의 입장에서 제품이 이해하기 쉽고, 사용하기 편하며, 심리적 마찰이 적은지 검수한다.

### 주요 검수 질문
- 처음 들어온 유저가 무엇을 해야 하는지 바로 아는가?
- 가입, 핵심 객체 생성, 초대/공유, 주요 액션 흐름이 귀찮지 않은가?
- 유저가 실수했을 때 복구할 수 있는가?
- 초보자도 제품에서 쓰는 용어를 이해할 수 있는가?
- 빈 화면에서 다음 행동을 안내하는가?
- 사용자의 감정적 마찰이 적은가?

### 도메인 적용 가이드
- 제품이 도입한 고유 용어(도메인 명칭)를 유저가 바로 이해할 수 있는가?
- 핵심 액션을 수행하기 전에 강제하는 사전 단계가 자연스러운가?

---

## 3. QA / Test Engineer View

### 역할
PRD의 요구사항이 실제로 테스트 가능한 수준으로 정의되어 있는지 검수한다.

### 주요 검수 질문
- 각 기능의 성공 조건이 명확한가?
- 실패 케이스가 정의되어 있는가?
- 권한 없는 접근, 입력 실패, 중복 요청 같은 예외가 있는가?
- Acceptance Criteria가 있는가?
- 테스트 시나리오로 바로 바꿀 수 있는가?
- 모호한 표현이 없는가?

### 나쁜 요구사항 예시
> 핵심 기능이 잘 동작해야 한다.

### 좋은 요구사항 예시
> 사용자는 정의된 형식(예: JPG/PNG/CSV/JSON 등)의 입력을 처리할 수 있다.
> 입력 단위당 최대 크기/길이 제한이 명시되어 있다.
> 실패 시 재시도 또는 복구 동작이 정의되어 있다.
> 처리 중에는 진행 상태가 사용자에게 표시된다.

---

## 4. Data Architect View

### 역할
데이터 구조가 제품의 핵심 기능, 확장성, 삭제 정책, 권한 정책을 안정적으로 지원할 수 있는지 검수한다.

### 주요 검수 질문
- 핵심 엔티티가 명확한가?
- 주요 엔티티 간 관계(소유/소속/참조)가 자연스러운가?
- 데이터 소유권이 명확한가?
- 삭제 정책(soft/hard, cascade 범위)이 정의되어 있는가?
- 향후 확장(검색, 공유, 결제, 분석 등)을 붙일 수 있는가?
- 데이터 중복이나 애매한 상태가 생기지 않는가?

### 도메인 적용 가이드
- 핵심 리소스의 소유자는 개인인가, 그룹인가, 조직인가?
- 사용자가 그룹/조직에서 빠지면 그가 생성한 리소스는 어떻게 처리되는가?
- 상위 엔티티가 삭제될 때 하위 엔티티가 함께 삭제되는가, 보존되는가?

---

## 5. DevOps / Infrastructure View

### 역할
서비스가 안정적으로 배포, 운영, 모니터링, 복구될 수 있는지 검수한다.

### 주요 검수 질문
- 배포 구조가 단순한가?
- 개발, 스테이징, 프로덕션 환경이 나뉘는가?
- 핵심 트래픽/처리량을 감당할 수 있는가?
- 로그와 모니터링 계획이 있는가?
- 장애가 나면 어디서 확인할 수 있는가?
- 백업과 복구 전략이 있는가?
- 환경변수와 시크릿 관리가 안전한가?

### 도메인 적용 가이드
- 트래픽 피크가 몰릴 때 시스템이 어떻게 스케일/큐잉되는가?
- 외부 의존성(스토리지, DB, 외부 API) 장애 시 격리/대체 전략이 있는가?
- 비동기 작업 실패를 어떻게 재시도/복구하는가?

---

## 6. Cost / Business View

### 역할
서비스 운영 비용이 통제 가능한지, 사용량 증가가 비즈니스 모델과 연결되는지 검수한다.

### 주요 검수 질문
- 무료 유저에게 어디까지 제공할 것인가?
- 사용자/그룹 단위 리소스 한도가 있는가?
- 원본/변형/캐시 등 복수 형태의 데이터를 모두 보관할 것인가?
- 사용자가 늘수록 비용이 어떻게 증가하는가? (선형/지수)
- 유료 플랜으로 전환할 기준이 있는가?
- 결제를 나중에 붙이기 쉬운 구조인가?

### 도메인 적용 가이드
- Free tier의 한도(저장량/호출량/시트 수 등)는 무엇으로 정의할 것인가?
- 비용을 가장 크게 유발하는 자원은 무엇이며, 이를 유료화로 묶을 수 있는가?
- 고비용 기능(AI 호출, 외부 API 등)을 크레딧/사용량 기반으로 둘 것인가?

---

## 7. Legal / Privacy View

### 역할
법적 문제, 개인정보 처리, 데이터 권리, 이용약관과 개인정보처리방침 관점에서 PRD를 검수한다.

### 주요 검수 질문
- 수집/저장하는 데이터에 개인정보 또는 민감정보가 포함되는가?
- 메타데이터(위치, 디바이스, 식별자 등)를 저장하거나 노출할 것인가?
- 사용자가 데이터를 삭제하면 실제 저장본도 삭제되는가?
- 탈퇴 시 데이터는 어떻게 처리되는가? (보존 기간/익명화/완전 삭제)
- 유저 데이터를 AI 학습 등 2차 용도로 사용하지 않는다는 정책이 있는가?
- 개인정보처리방침과 이용약관이 필요한가? 어디까지 다루는가?

### 도메인 적용 가이드
- 업로드 데이터에서 민감 메타데이터를 기본적으로 제거할 것인가?
- 그룹 내에서 공유된 데이터의 소유권은 누구에게 있는가?
- 사용자가 삭제를 요청하면 원본, 파생본, 캐시까지 일관되게 삭제되는가?

---

## 8. Growth / Onboarding View

### 역할
처음 들어온 유저가 빠르게 가치를 경험하고, 초대/공유를 통해 자연스럽게 확산될 수 있는지 검수한다.

### 주요 검수 질문
- 처음 들어온 유저가 1분 안에 가치를 느끼는가?
- 첫 번째 핵심 액션까지의 마찰이 적은가?
- 초대/공유 흐름으로 자연스럽게 이어지는가?
- 초대받은 사람이 가입하도록 유도되는가?
- 빈 화면이 다음 행동을 유도하는가?
- 리텐션을 만들 수 있는 반복 사용 이유가 있는가?

### 도메인 적용 가이드
- 초대 링크를 받은 사람이 로그인 후 원래 맥락(특정 리소스/화면)으로 정확히 돌아오는가?
- 첫 핵심 액션 완료 직후 “다음 협업/공유 단계”로 자연스럽게 연결되는가?

---

## 9. Accessibility / Inclusive Design View

### 역할
다양한 사용자, 기기, 환경에서도 제품을 사용할 수 있는지 검수한다.

### 주요 검수 질문
- 모바일에서 조작하기 쉬운가?
- 텍스트 대비가 충분한가?
- 버튼/액션 이름이 명확한가?
- 키보드 접근성이 필요한 화면이 있는가?
- 비텍스트 콘텐츠(이미지, 차트 등)에 대체 텍스트나 설명이 필요한가?
- 고령자나 비전문 사용자도 쓸 수 있는가?

### 도메인 적용 가이드
- 자주 쓰이는 주요 액션 버튼이 충분히 크고 명확한가?
- 로딩/실패 상태에 대체 표시가 있는가?
- 색상에만 의존해 상태를 구분하지 않는가?

---

## 10. Competitive Analysis / Competitor Research View

### 역할
경쟁사와 대체재를 기준으로 이 제품이 왜 필요한지, 어디에서 차별화되는지, 어떤 기능을 따라가거나 버려야 하는지 검수한다.

### 주요 검수 질문
- 사용자는 현재 이 문제를 어떤 서비스로 해결하고 있는가?
- 직접 경쟁사와 간접 대체재가 구분되어 있는가?
- 경쟁사 대비 이 제품의 핵심 차별점은 무엇인가?
- 기존 서비스가 이미 잘 해결하는 영역을 불필요하게 다시 만들고 있지 않은가?
- 경쟁사가 제공하지만 MVP에서 의도적으로 제외할 기능은 무엇인가?
- 경쟁사가 해결하지 못한 불편이나 틈새는 무엇인가?
- 가격, 성능, 사용성, 협업 방식, 통합/확장성에서 비교 우위가 있는가?

### 도메인 적용 가이드
- 직접 경쟁사: 같은 문제를 같은 방식으로 푸는 제품
- 간접 대체재: 다른 카테고리지만 사용자가 현재 우회 수단으로 쓰는 제품
- 비-소비: 아무것도 쓰지 않고 문제를 견디는 유저층
> 위 세 축을 모두 매핑한 뒤, 이 제품이 어떤 축에서 우위를 만들지 한 문장으로 정의한다.

### 산출물
- 경쟁사 매트릭스
- 대체재 분석
- 차별화 포인트
- 따라 하지 않을 기능 목록
- MVP에서 반드시 이겨야 하는 사용 시나리오

---

## 11. Security / Threat Modeling View

### 역할
전체 공격 표면과 보안 리스크를 식별하고, PRD 단계에서 위험한 흐름을 사전에 막는다.

### 주요 검수 질문
- 이 기능이 어떤 공격 표면을 만드는가?
- 로그인하지 않은 유저가 접근할 수 있는 경로가 있는가?
- 권한 없는 사용자가 URL 직접 입력으로 접근할 수 있는가? (IDOR)
- API 요청을 조작하면 다른 사용자/그룹 데이터에 접근할 수 있는가?
- 민감한 정보가 로그, URL, 응답값에 노출되는가?
- 클라이언트에서 숨긴 정보가 서버에서 다시 검증되는가?

### 도메인 적용 가이드
- 리소스 식별자(예: `:id`)를 URL에 직접 넣었을 때 권한 검증이 일어나는가?
- 공유 링크/토큰이 외부에 유출되면 어디까지 접근 가능한가? (만료/회수 가능 여부)
- 삭제된 리소스의 직접 URL이 계속 살아 있지 않은가?

---

## 12. Authorization / Permission View

### 역할
누가 어떤 데이터와 기능에 접근할 수 있는지, 권한 정책이 명확하고 일관적인지 검수한다.

### 주요 검수 질문
- 누가 그룹/워크스페이스를 만들 수 있는가?
- 누가 멤버를 초대할 수 있는가?
- 누가 리소스를 생성/수정/삭제할 수 있는가?
- 누가 어떤 데이터를 조회할 수 있는가?
- 멤버가 그룹에서 나가면 기존 리소스 접근 권한은 어떻게 되는가?
- 공유 링크를 통해 접근한 사용자는 어디까지 볼 수 있는가?
- DB 권한과 외부 자원(스토리지/캐시 등) 권한이 일관되는가?

### 도메인 적용 가이드
- owner / admin / member / viewer / shared-link 등 역할별 권한이 명시되어 있는가?
- 그룹 외부 사용자가 외부 자원 URL만으로 데이터에 접근할 수 없는가?
- 리소스 단위 공유와 그룹 단위 공유가 서로 충돌하지 않는가?

---

## 13. Abuse Prevention View

### 역할
서비스를 악용하거나 남용하는 사용자를 막고, 스팸, 비용 공격, 리소스 낭비를 방지하는지 검수한다.

### 주요 검수 질문
- 무료 유저가 무한히 리소스를 생성할 수 있는가?
- 초대/공유 링크를 스팸처럼 보낼 수 있는가?
- 한 유저가 그룹/워크스페이스를 무한 생성할 수 있는가?
- 대용량 입력으로 비용 공격이 가능한가?
- 악성/비정상 입력을 막는가?
- 공유 링크가 검색엔진에 노출될 위험이 있는가?
- rate limit이나 quota가 정의되어 있는가?

### 도메인 적용 가이드
- 유저/그룹당 일일·월간 사용량 한도가 있는가?
- 초대/공유 링크 생성 횟수 제한이 있는가?
- 비정상적으로 큰 입력이나 위장된 파일을 차단하는 검증이 있는가?

# 추천 PRD 검수 순서

```text
PRD Draft
→ Product / Strategy 검수
→ Competitive Analysis 검수
→ User / UX 검수
→ Data / Engineering 검수
→ Security / Permission 검수
→ Legal / Privacy 검수
→ QA 검수
→ Cost / Business 검수
→ Growth / Onboarding 검수
→ 최종 MVP Scope 확정
```

# 핵심 정리

좋은 PRD 검수는 “이 기능을 만들 수 있는가?”만 확인하는 것이 아니다.
좋은 PRD 검수는 아래 질문을 동시에 확인해야 한다.

```text
- 이 제품은 왜 필요한가?
- 사용자가 실제로 편하게 쓸 수 있는가?
- 경쟁사 대비 차별점이 있는가?
- 데이터 구조가 오래 버틸 수 있는가?
- 보안과 권한이 안전한가?
- 비용이 통제 가능한가?
- 테스트 가능한 수준으로 요구사항이 명확한가?
- 출시 후 성장과 운영이 가능한가?
```
