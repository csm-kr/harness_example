# 운영 Runbook

장애·이상 상태에서 **무엇을 어떤 순서로 확인하고 조치하는지**의 절차서.
새로운 막힘이 생기면 즉시 한 항목 추가.

## 알림 채널
- 1차: {예: Slack #data-pipeline-alerts}
- 2차 (오프 시간): {예: PagerDuty 로테이션}
- 모니터링 대시보드: {예: Grafana 링크 / Datadog 링크}

---

## 증상 1: 일 배치가 SLA 시간(예: 03:00 KST)까지 완료 안 됨

### 재현 / 확인
1. 오케스트레이터 UI 에서 해당 DAG 실행 상태 확인
2. 실패 단계 ({extract / transform / load}) 식별
3. 로그 확인: 오케스트레이터 로그 + `runs/{ts}/` 또는 컨테이너 로그

### 흔한 원인
- 소스 DB 부하 → extract 가 lock wait
- 데이터 양 폭증 → transform OOM
- 싱크 쪽 권한 만료 → load 401/403

### 조치
- 일시적 재시도: 단계만 retry (전체 DAG 가 아님)
- OOM: 워커 메모리 증설 / 쿼리 분할
- 권한: 자격증명 갱신 후 재실행
- SLA 미달 통보: {담당자 / 다운스트림 사용자}

---

## 증상 2: DLQ 적재 폭증 (입력 계약 위반)

### 재현 / 확인
1. quarantine 테이블 또는 DLQ 큐 건수 추이 확인
2. 위반 패턴 분석 — 어떤 컬럼·어떤 값에서 깨졌는가

### 흔한 원인
- 소스 측 스키마 변경 (운영팀이 사전 통보 없이 컬럼 추가/타입 변경)
- 새 enum 값 출현
- 시간대 / 인코딩 문제

### 조치
- 단기: 위반 레코드 격리 유지, 통보 → 소스 측 합의
- 영구: [DATA_CONTRACTS.md](./DATA_CONTRACTS.md) 의 계약 갱신 → 스키마 코드 갱신 → 재처리

---

## 증상 3: 출력이 갑자기 중복

### 재현 / 확인
1. 싱크에서 유니크 키 기준 중복 행 수 측정
2. 최근 적재 로그에서 같은 키 여러 번 적재되었는지

### 흔한 원인
- watermark / cursor 가 갱신 안 됨 (멱등성 깨짐)
- 같은 트리거가 동시 실행 (DAG 중복 스케줄)

### 조치
- 즉시 중복 제거 SQL (백업 후 실행)
- 원인 분석: cursor 저장 위치 확인, 트리거 동시성 차단 (오케스트레이터 max_active_runs)

---

## 증상 4: 다운스트림 (대시보드/ML) 이 데이터 못 봄

### 재현 / 확인
1. 싱크 자체에는 데이터 있는지 (dt 파티션 검사)
2. 다운스트림 측 캐시·뷰 갱신 여부

### 조치
- 데이터 자체가 없음 → 위 증상 1·2·3 재진단
- 데이터는 있으나 다운스트림 미반영 → 다운스트림 캐시 무효화

---

## 사후 (Postmortem)
P1 이상 사고는 24시간 이내 한 페이지 사후 작성:
- 무엇이 일어났는가 (timeline)
- 사용자에게 미친 영향
- 근본 원인
- 재발 방지 조치 (코드/프로세스)
- 결정으로 굳어지면 [ADR.md](./ADR.md) 에 항목 추가
