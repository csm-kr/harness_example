# 출시 절차

## 버전 정책
- 형식: {예: SemVer (X.Y.Z) — major: 호환 안 되는 변경, minor: 기능, patch: 버그}
- 빌드 번호: 매 빌드마다 +1 (스토어 업로드 단위로 unique)
- 정식 vs 베타 트랙: {예: 정식 — main 브랜치 / 베타 — beta 브랜치}

## 출시 전 체크리스트
- [ ] PRD 의 "성공 기준" 메트릭이 회귀하지 않음 ([PRD.md](./PRD.md) 참고)
- [ ] [PUSH.md](./PUSH.md) 의 푸시 통합이 양쪽 플랫폼 동작
- [ ] 충돌(crash) 보고 0건 — 직전 24시간 dogfood
- [ ] 스토어 메타데이터 (스크린샷·설명·키워드) 갱신
- [ ] 개인정보 처리방침 / 이용약관 변경 사항 반영
- [ ] 환경별 .env: 운영 키로 빌드되었는지 재확인 (개발 키 금지)

## iOS — App Store
- 빌드: {예: Xcode + fastlane / EAS Build}
- 업로드: {예: TestFlight 먼저 → 외부 테스터 → 정식}
- 심사 평균 소요: 1~2일 (거절 사유 다발: 개인정보 권한 사용 명분 부족, 결제 외부 링크)
- 크래시 심볼: dSYM 자동 업로드 설정 필요

## Android — Google Play
- 빌드: {예: AAB (Android App Bundle) / fastlane / EAS Build}
- 트랙: 내부 테스트 → 비공개 테스트 → 공개 테스트 → 정식
- 심사 평균 소요: 수 시간 ~ 1일
- 정책 위반 다발: 권한 명분 부족, 데이터 안전성 양식 누락

## 롤백
- iOS: 새 빌드를 즉시 다시 올리는 것 외에 롤백 없음 (이전 빌드 재출시 불가)
- Android: 직전 버전 다시 staged rollout 으로 푸시 가능
- 핫픽스 정책: {예: P0 버그는 24시간 내 patch 빌드 출시}

## 변경 이력
- 출시 시 `CHANGELOG.md` 갱신 (선택)
- 사용자에게 보일 변경 사항: 스토어 "이번 버전" 한국어 작성
